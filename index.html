Dynamic Tables Eviltester

Page tested: https://testpages.eviltester.com/styled/tag/dynamic-table.html

JS Snippet: eviltester_dynamic_table.js

Testcases

Table of human readable testcases: Google Sheet

Testcases included in code as array of objects

Testcase Structure
  • name: identifier of testcase, string
  • caption: value for caption of new table, string
  • tableID: value for id of new table; must be used to address table in tests, string
  • tabledata: array of row objects, attribute names=column names, values=cell content
  • expected: object with attributes and values to assert against

How to run test 

runTest(testcase)

Examples:

runTest(testcases[0])
runTest(generateTestcase(columns, rows))

Locator strategy

Input element IDs are used as locators.

The ID of the table changes according to the tableID provided in the testcase.
testcase.tableID is therefore used to address the table for assertions.

Function Description

runTest(testcase)
  1. console.logs the testcase name
  2. calls enterData
  3. calls assertTable with setTimeout to ensure table generation is finished before assertion is executed
enterData(testcase)
  1. Sets the values of the input fields using document.querySelector(attribute.locator).value = testcase.attribute
  2. JSON.stringify used to convert testcase.tabledata to target format.
  3. Clicks the Refresh button
assertTable(testcase)
  • Asserts the ID and caption of the generated table.
  • Current assertions for table content are number of rows and columns.
  • Expected number of columns is calculated from the number of unique attribute names in the tabledata array.
  • Cell content was verified by visual inspection.
  • For generated testcases, each cell has the x,y coordinates of the cell as value.
generateTestcase (rows, columns)

Helper function to ensure testcase with generated tabledata has the right structure.

myGridHelper(columns, rows)
  • Helper function to generate the tabledata for testing large tables.
  • Simple for-loop to generate the required number of column names.
  • 2 nested for-loops to generate the rows and cell content.
  • Cell content are the x,y coordinates of the cell in the table. This makes cell content verification easy.

Dynamic Content

Page tested: https://moatazeldebsy.github.io/test-automation-practices/?#/dynamic-elements

JS-Snippet:dynamicContentTesting.js

Selenium IDE:

Testcases

  1. Toggle Hide-Button from Hide to Show: Content is displayed
  2. Toggle Hide-Button from Show to Hide: Content is not displayed
  3. Testcases for various slider element positions

Slider positions were set manually because React Apps don't allow anonymous data manipulation from the console.

How to Run Test

Toggle Show/Hide (Testcases 1, 2)
checkHideShow()
Check Time till Dynamic Content Appears (Testcases 3)
  1. Set slider position
  2. Run test
checkDynamicContent(times)

Locator Strategy

  • For elements that exist on page load, constants for the nodes were used. The maintainer of the page provides the elements with data-test attributes that allow easy identification.
  • For dynamically generated elements, the parent element's first child was determined at runtime.

Function Description

determineExpected()

Checks the classList.length of the hidable element and returns the opposite using a ternary expression. This is a lazy check because the tested element had no other classes. In real life, checking for classList.contains('hidden') would be the better choice.

checkDynamicContent(times = 10)

checkDynamicContent uses a setInterval of 500 ms to check if the loading indicator is still present. The check stops after times*500ms. Each interval console logs the time passed and the status of the loading indicator. Assertion of time passed until UL appears against value set in slider.
