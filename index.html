Testcase Structure

Testcases is a list of testcase objects.

Testcase Object Description:

  • name: testcase identifier, string
  • field1: input data for first input field, number or string
  • field2: input data for second input field, number oder string
  • operator: text to be selected in the function selection list, string
  • expected: Expected result, number or string
  • description: short description of test case, string
Basic Testing with Chai Assertion
How to Run Test

Run calculator.js to execute all tests.

Locator Strategy

IDs are used as locators.

Function Description
runAllTests(testcases)

The function loops through testcases and calls test(testcase) for each.

Each test is run in its own browser instance.

test(testcase)

The function generates the driver and browser instance for the test case and calls the function fillFieldsAndCheck(driver, testcase) executing data input and assertion.

fillFieldsAndCheck(driver, testcase)

This function executes the test case and asserts the result:

  1. Calls fillField(driver, fieldId, keys ) for each field
  2. Calls doSelect(driver, operator) to set the correct operator
  3. Clicks on the Calculate button
  4. Asserts the result
fillField(driver, fieldId, keys)

This function fills the field and asserts the field value after input.

doSelect(driver, operator)

This function creates a Select object to interact with the selection list.

  1. Creates Select object named 'select'
  2. Uses select.selectByVisibleText(operator) to set option
  3. Gets the text of the selected option using select.getFirstSelectedOption() followed by .getText() on the result
  4. Asserts selected option matches operator
With Mocha

Now using mocha as a test framework and a page class to encapsulate the interactions with the app.

How to Run Test
  • Make sure youre working directory is ./Classes/Tests.
  • Run npx mocha ./Calculator/calculator_mocha.js
Classes and Methods
Generator class (imported)

Reusing the random test case idea from the first cycle, this helper class generates a test set for all strinctly numeric test cases with random numbers.
Resulting test set: randomTC ist used in the thrid mocha test.

Calculator class

Encapsulates all functions to interact with the app

  • async open(): Initializes the Selenium Webdriver with the page url
  • asny resize(): Resizes the browser window to a defines size.
  • async isPresent(locator): Checks if the element specified by the locator is found. Returns true or false
  • async setInput(locator, input): Locates the input element specified by the locator, empties the input field and fills in input usinge sendKeys()
  • async selectInList(locator, input): Locates the selection list specified by locator and selects the input by text using Selenium's Selectclass.
  • async click(locator):Try/Catch around the function to click the element specified by locator. The catch helps with troubleshooting if elements are not clickable for some reason.
  • async setAllInputs(testcase): Wrapper function to fill both fields, set the select list and click the button.
  • async getAnswer():Retrives the calculator result for assertion. Uses implicit wait until.elementIsVisible.
  • async getTitle():Retrives the page title for assertion.
  • async close(): Method to gracefully close the driver after a test.
Mocha test suite

URL of test page is passed to the new calculator object.

Individual browser instance is opened and resized to 1504 x 1024 px for each test and closed after the test.

Test 1: test page title and input fields

Simply asserts the page title and checks if all gui elements required for the tests are found.

Test2: ForEach run for the set of manually defined testcases including non-numerical inputs

The manually defined testcases match the test cases defined in the google sheet. They are imported as testcases from the testdata_calculator.js file also used by the older test.
The calculation result retrieved with await calculator.getAnswer() is asserted against the manually defined expected result. Expected results are rounded.

Test 3: ForEach for random number test set

The calculation result retrieved with await calculator.getAnswer() is asserted against the expected result calculated from the random input. Expected results are not rounded. This leads to assertion errors in the division testcases.
Chai assert cannot be used with chai-almost().
ToDo: Include rounding into Generator class.

Dynamic Tables Eviltester

Page tested: https://testpages.eviltester.com/styled/tag/dynamic-table.html

JS Snippet: eviltester_dynamic_table.js

Testcases

Table of human readable testcases: Google Sheet

Testcases included in code as array of objects

Testcase Structure
  • name: identifier of testcase, string
  • caption: value for caption of new table, string
  • tableID: value for id of new table; must be used to address table in tests, string
  • tabledata: array of row objects, attribute names=column names, values=cell content
  • expected: object with attributes and values to assert against
How to run test 
runTest(testcase)

Examples:

runTest(testcases[0])
runTest(generateTestcase(columns, rows))
Locator strategy

Input element IDs are used as locators.

The ID of the table changes according to the tableID provided in the testcase.
testcase.tableID is therefore used to address the table for assertions.

Function Description
runTest(testcase)
  1. console.logs the testcase name
  2. calls enterData
  3. calls assertTable with setTimeout to ensure table generation is finished before assertion is executed
enterData(testcase)
  1. Sets the values of the input fields using document.querySelector(attribute.locator).value = testcase.attribute
  2. JSON.stringify used to convert testcase.tabledata to target format.
  3. Clicks the Refresh button
assertTable(testcase)
  • Asserts the ID and caption of the generated table.
  • Current assertions for table content are number of rows and columns.
  • Expected number of columns is calculated from the number of unique attribute names in the tabledata array.
  • Cell content was verified by visual inspection.
  • For generated testcases, each cell has the x,y coordinates of the cell as value.
generateTestcase (rows, columns)

Helper function to ensure testcase with generated tabledata has the right structure.

myGridHelper(columns, rows)
  • Helper function to generate the tabledata for testing large tables.
  • Simple for-loop to generate the required number of column names.
  • 2 nested for-loops to generate the rows and cell content.
  • Cell content are the x,y coordinates of the cell in the table. This makes cell content verification easy.

Page tested: https://moatazeldebsy.github.io/test-automation-practices/?#/dynamic-elements

Testcases

  2. Toggle Hide-Button from Show to Hide: Content is not displayed
  3. Testcases for various slider element positions

Slider positions were set manually because React Apps don't allow anonymous data manipulation from the console.

How to Run Test
Toggle Show/Hide (Testcases 1, 2)
checkHideShow()
Check Time till Dynamic Content Appears (Testcases 3)
  1. Set slider position
  2. Run test
checkDynamicContent(times)
Locator Strategy
  • For elements that exist on page load, constants for the nodes were used. The maintainer of the page provides the elements with data-test attributes that allow easy identification.
  • For dynamically generated elements, the parent element's first child was determined at runtime.
Function Description
determineExpected()

Checks the classList.length of the hidable element and returns the opposite using a ternary expression. This is a lazy check because the tested element had no other classes. In real life, checking for classList.contains('hidden') would be the better choice.

checkDynamicContent(times = 10)

checkDynamicContent uses a setInterval of 500 ms to check if the loading indicator is still present. The check stops after times*500ms. Each interval console logs the time passed and the status of the loading indicator. Assertion of time passed until UL appears against value set in slider.

Eviltester Triangle (Cycle 1)

Page tested: https://testpages.eviltester.com/styled/apps/triangle/triangle001.html

JS Snippet: testscript_triangle_app.js

Testcases

Table of human readable testcases: Google Sheet

Testcase Structure
  • Each test case is run with several sets of data for robustness.
  • Test data generated with ChatGPT and matched using generateTestSet.
  • Special test sets for "Not A Number" and very large numbers written by hand.
  • Final testSets are arrays of objects:
    • input: array of 3 values for triangle sides
    • output: expected answer of identify-triangle-action
How to Run Test

Run the function with a test set of your choice, e.g. testSetEqui:

testTestCase(testSetName)
Locator Strategy

Input element IDs are used as locators. Field IDs are generated from the base "side" in the loop calling typeInField.

Function Description
generateTestSet(tcInput, tcOutput)
  • Helper function for test set generation
  • Loops through input data and assigns expected output
typeInField(fieldID, fieldData)
  1. Puts fieldData in the input field
  2. Asserts the value of the input field
clickButton(buttonID)
  • Clicks the button given as parameter
checkTriangle(testcase)
  1. Loops through input data array and calls typeInField for each
  2. Calls clickButton
testTestCase(testCaseSet)
  1. Loops through the testCaseSet and calls checkTriangle for each
  2. Asserts the output
Lessons Learned
  • Always make sure your assertion is robust!
  • Running testCaseSets for T002 - T006 seemed to work, but T007 (NaN) failed, and T008 failed sometimes.
  • Reason: Incorrect use of setTimeout and identical expected results in T002-T006 masked the issue.
